# kchiro Directory Structure

This document outlines the organization and purpose of each directory in the `kchiro` workspace.

```
kchiro/
├── backend/       # FastAPI web server and backend business logic
├── frontend/      # React single-page application (UI/UX)
├── blender/       # Blender interface and execution harness
├── generators/    # Procedural mesh generation scripts using the Blender Python API (bpy)
├── schemas/       # JSON and Pydantic validation schemas for asset specifications
├── exports/       # Output folder for generated 3D assets (.glb files)
├── renders/       # Output folder for rendered preview images (PNG files)
├── database/      # SQLite database files and database models
└── docs/          # Project documentation, guides, and specifications
```

---

## Folder Explanations

### `backend/`
Contains the FastAPI server code. It handles HTTP requests from the frontend, routes them to the prompt processor, executes background generation tasks, queries the SQLite database, and serves the static files (GLB exports and PNG renders).

### `frontend/`
Contains the React application code. This is where the user enters text prompts, views preview renders, inspects generated asset details, and downloads the finalized GLB models.

### `blender/`
Contains the Blender integration harness. This module handles executing scripts in Blender's headless mode (`blender --background`), passing inputs via command line arguments or temporary files, and capturing output logs.

### `generators/`
Contains the core procedural geometry engines. Each file here defines how to build specific shapes (e.g., `sword.py`, `table.py`, `barrel.py`, `crate.py`) using the Blender Python API (`bpy`). These files are executed *inside* the Blender environment.

### `schemas/`
Contains the data models and schemas defined using Pydantic. These schemas ensure that the structured output generated by Ollama conforms strictly to the parameters expected by our procedural generators.

### `exports/`
The storage folder for generated 3D files. When a model is successfully generated and textured in Blender, it is saved here as a `.glb` (glTF binary) file, ready for web-previewing or direct use in game engines.

### `renders/`
The storage folder for 2D previews. After exporting the `.glb`, Blender positions a camera, sets up a default three-point lighting rig, and renders a high-quality PNG preview of the model. These previews are served to the frontend.

### `database/`
Contains the SQLite database storage. The database tracks the history of generation tasks, user prompts, extracted JSON specifications, file paths for exports/renders, and generation timestamps.

### `docs/`
Contains project documentation, notes, tutorials on Blender Python scripting, API guides, and future roadmap files.
